# AI 开发协作规则

本文件定义 AI 在本项目中协助开发、修改代码、更新文档、调整配置、编写测试和执行脚本时必须遵守的规则。  
在修改代码、文档、配置、提示词、测试或脚本之前，必须先阅读并遵守本文件。

## 核心原则

1. 先理解需求、现有实现、模块边界和影响范围，再进行修改。
2. 优先做当前任务所需的最小改动。
3. 不重写、不删除、不绕过、不重构与当前任务无关的稳定逻辑。
4. 每个改动都必须能说明：为什么需要、影响哪里、如何验证。
5. 如果第一次修复不充分，必须重新分析根因，再做第二次修改；不要盲目试错式补丁。

## 必须文档与可选文档

当项目中存在 `AGENTS.md` 时，必须区分“项目必备文档”和“按需维护文档”。

项目建议具备的基础文档：

1. `AGENTS.md`：AI 开发规则、安全边界和协作约束。
2. `README.md`：项目用途、安装方式、常用命令、主要工作流。
3. `REQUIREMENTS.md` 或等价需求文档：业务需求、产品规则、验收标准。

按需维护的文档：

1. `DEV_STANDARDS.md`：当项目有明确代码规范、架构规范、测试规范时维护。
2. 部署说明文档：当项目需要服务器部署、定时任务、后台运行或固定更新流程时维护。
3. 模块级 README：当某个模块可以独立使用，或复杂到需要单独说明时维护。
4. `docs/` 下的交接文档：当阶段上下文较多，或需要跨窗口、跨人员、跨 Agent 交接时维护。

首次进入项目，或用户要求同步、收尾、整理、交接时，应检查基础文档是否存在。  
如果缺少基础文档，应告知用户缺失项；当前任务确实需要时，可创建简版。  
不要因为本规则提到某个可选文档，就强行创建无意义文件。

## 权威文档与同步规则

1. 需求文档是业务规则和产品行为的权威来源。
2. 业务规则变化时，应先更新或同步需求文档，再修改代码或同时修改。
3. 新增功能、改变行为、增加命令、调整配置后，应检查 README、需求文档、部署说明、配置示例和模块文档是否需要同步。
4. 新配置项必须写入相关配置模板或示例，并说明默认值和作用。
5. 如果项目使用提示词文件，提示词变化应使用版本化文件，不覆盖旧版本；同时更新当前版本索引文件。
6. 如果文档和代码冲突，应先检查代码确认当前真实行为，再根据用户意图决定是修代码还是修文档。

## 模块边界

1. 先理解已有模块职责，再决定修改位置。
2. 不把业务判断、数据持久化、通知、外部 API、UI、测试工具等职责混到无关模块中。
3. 新增逻辑应放到最符合职责的模块中。
4. 如果必须跨多个模块修改，应说明调用链和每个文件为什么必须改。
5. 不为了快速实现而绕过已有架构边界。
6. 不在只读模块中加入写操作，不在通知模块中加入业务决策，不在规则模块中调用 AI 或外部服务，除非项目架构明确允许。

## 数据安全

1. 默认不修改生产数据、线上数据、外部系统数据。
2. 不写入、删除、更新外部业务数据库，除非用户明确要求且项目允许。
3. 不读取、输出、提交或写入真实密钥、密码、Token、Cookie、Webhook、私钥等敏感信息。
4. `.env`、本地密钥文件、运行日志、数据库文件、缓存、虚拟环境、备份目录等运行时数据，默认不得修改或提交。
5. 修改数据结构、索引、迁移脚本前，必须评估兼容性和回滚影响。
6. 涉及身份、唯一键、去重、幂等逻辑时，必须确认使用完整业务主键，不偷懒使用局部 ID。

## 恢复与回滚

1. 不要从备份目录整体覆盖源码、配置、文档或提示词，除非当前文件不可读且影响已确认。
2. 恢复前必须比较当前文件和备份文件，优先做局部修复。
3. 不使用破坏性命令重置用户改动，除非用户明确要求。
4. 如果工作区已有他人或用户改动，不要回滚；应在现有改动上继续工作。
5. 恢复后必须运行相关测试，并确认没有丢失较新的已接受改动。

## 规则与 AI 判断

1. 硬规则只应包含客观、确定、AI 不应覆盖的事实。
2. 软规则、语义判断、上下文判断，应允许 AI 或人工复核。
3. 不随意把软规则升级为硬失败。
4. AI 可以覆盖非硬规则的误判，但不能覆盖客观硬失败。
5. 如果 AI 覆盖了规则结果，最终输出必须清晰，不保留会误导用户的旧失败原因。

## 通知与外部集成

1. 通知发送、消息格式、频率限制、冷却时间，应遵守配置。
2. 不在通知模块中写业务决策。
3. 外部接口调用失败时，应有可恢复或可降级策略。
4. 对可能刷屏、重复发送、误通知的逻辑，必须验证幂等和限流。
5. 外部系统联调应优先使用最小安全验证，不做破坏性探测。

## 学习、标注与反馈数据

1. 标注词汇和状态必须在文档、报表、训练/学习逻辑中保持一致。
2. 人工标注数据是学习依据，不能静默丢弃。
3. 标注文件被占用或写入失败时，应有队列、重试或明确告警。
4. 标注语义变化时，应同步更新读取、写入、报表、学习和文档。
5. AI 学习或提示词优化应可追踪来源和版本。

## 自测规则

1. 每个新功能必须自测后才算完成。
2. 每个 bug 修复必须验证失败场景和至少一个相关非回归场景。
3. 规则逻辑变化应增加或更新规则测试。
4. 涉及通知、Excel、SQLite、调度、提示词、外部数据解析、AI 调用的改动，应在最小安全范围内验证相关流程。
5. 无法运行测试时，必须说明原因、阻塞点和剩余风险。
6. 完成后必须说明测试命令和结果。

## 独立调试规则

1. 新需求、新功能、不清楚的外部集成、不确定的数据源行为，应先用独立调试脚本或最小测试验证，再改生产代码。
2. 调试脚本必须聚焦，只验证具体问题，不修改生产数据。
3. 未验证清楚前，不把临时调试逻辑合入正式代码。
4. 调试完成后删除临时文件，除非它被明确提升为维护工具或正式测试。
5. 如果因为缺少凭证、网络、环境或依赖无法验证，应报告阻塞，不要猜测后直接修改生产代码。

## 临时文件规则

1. 临时调试文件、一次性脚本、临时输出、手工测试文件，调试完成后必须删除。
2. 不留下 `debug_*`、`tmp_*`、`test_manual_*` 等临时文件，除非明确提升为维护工具。
3. 如果临时脚本变成正式工具，必须补充用途和使用说明。

## 修改范围规则

1. 优先只修改当前任务需要的模块。
2. 不修改无关格式、注释、配置、提示词、测试。
3. 不为了让测试通过而削弱安全检查。
4. 不禁用审计、通知、队列、学习、持久化等核心逻辑，除非用户明确要求并记录影响。
5. 对大范围改动，应先说明方案和风险，再执行。

## 阶段收尾与知识同步

当用户表达阶段结束、需要整理、同步、交接、更新记忆或清理文档时，必须执行阶段收尾同步，而不是只给普通总结。

触发语包括但不限于：

1. `sync up`
2. `tidy up docs`
3. `update memory`
4. `clean up docs`
5. `/sync`
6. `/neat`
7. `同步一下`
8. `整理文档`
9. `整理一下`
10. `更新记忆`
11. `梳理一下`
12. `收尾`
13. `这个阶段做完了`
14. `新人能直接上手`

如果已有开发上下文，用户只说“整理”“收一下”“收尾一下”等，也应视为触发。

触发后应检查：

1. 本阶段代码变化。
2. 配置变化。
3. 命令变化。
4. 生成文件变化。
5. 测试覆盖。
6. 已知限制和后续风险。
7. README、需求文档、部署说明、模块文档、交接文档是否需要同步。

如果新增了命令或工作流，应记录：

1. 命令作用。
2. 何时使用。
3. 需要的配置或环境变量。
4. 输出文件。
5. 安全验证步骤。

如果服务器部署或更新流程变化，应更新部署说明；若项目没有部署需求，则跳过。  
如果阶段内容较多，应更新或创建 `docs/` 下的交接文档。

最终回复应包含：

1. 检查或修改了哪些文档。
2. 当前行为如何记录。
3. 做了哪些验证。
4. 仍有哪些风险或延期项。
5. 下一步建议。

阶段收尾同步不是大重构授权，不应借机做无关清理。

## 项目记忆与交接

当用户要求保存记忆、更新记忆、同步上下文、准备交接、换新窗口继续、让新人或其他 Agent 接手时，应把稳定上下文写入仓库文档，而不是依赖聊天历史。

可以保存的内容：

1. 当前阶段状态。
2. 已确定的设计决策及原因。
3. 已验证可用的命令。
4. 部署和更新步骤。
5. 重要配置开关和默认值。
6. 已知限制、延期问题、下一步计划。
7. 用户需要关注的数据文件或产物。
8. 开发过程中确认的模块边界和安全规则。

不要保存：

1. 密钥、密码、Token、Webhook、Cookie、私钥。
2. 已无意义的临时调试思路。
3. 运行数据、日志、缓存、虚拟环境、备份内容。
4. 未验证的猜测，除非明确标注为待确认问题。

推荐保存位置：

1. `AGENTS.md`：稳定 AI 开发规则。
2. `REQUIREMENTS.md` 或需求文档：业务规则和产品行为。
3. `README.md`：使用方式、命令、用户工作流。
4. 部署说明：服务器部署和更新操作。
5. 模块 README：独立工具或大型模块说明。
6. `docs/` 下的交接文档：阶段状态、待办、风险和恢复上下文。

交接文档应包含：

1. 已完成什么。
2. 当前关闭或有风险的功能。
3. 下一步应运行什么命令。
4. 哪些文件重要。
5. 哪些测试已经通过。
6. 哪些内容未经用户允许不能碰。

## 完成检查清单

完成开发任务前必须确认：

1. 改动符合本文件规则。
2. 没有修改无关文件。
3. 临时调试文件已删除。
4. 已运行合适的自测，或说明无法运行的原因。
5. 已检查文档、配置示例、部署说明是否需要更新。
6. 已总结修改文件、行为影响、验证结果和剩余风险。
